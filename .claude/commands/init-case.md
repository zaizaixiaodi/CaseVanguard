# /init-case — 初始化案件工作区

## 前置检查

1. 检查 `workspace/meta/case-state.json` 是否已存在。
2. 如果已存在且 `report_finalized` 为 false → 告知律师当前案件尚未完成，建议先执行 `/status` 查看进度，或 `/clean-workspace` 清理后再初始化。
3. 如果已存在且 `report_finalized` 为 true → 提示律师执行 `/clean-workspace` 清理后再初始化新案件。
4. 如果不存在 → 继续初始化流程。

## 步骤一：结构化引导

以自然对话方式向律师展示以下引导表，**不强制填满所有字段**，允许"暂不确定"或"后续补充"：

```
请提供以下案件信息（能填多少填多少，后续可随时补充）：

1. 案由：
2. 原告：
3. 被告（可多个）：
4. 简要案情：
5. 你目前的初步判断或假设（可选）：
6. 特别需要关注的点（可选）：
7. 策略性补充（可选，如"怀疑挂靠关系"、"重点关注免责条款"）：
```

等待律师回复后，解析律师的输入，填入对应字段。对于律师没有提供的字段，留空字符串或空数组。

## 步骤二：法律要件拆解

根据律师提供的案由（case_type），生成 `legal_elements_checklist`。以下是常见案由的要件拆解：

### 提供劳务者受害责任纠纷
- 主体适格：劳务关系是否成立
- 权利基础：受害人的合法权益
- 侵权事实：损害发生的经过
- 损害后果：伤情及损失金额
- 因果关系：劳务行为与损害的关联
- 过错/责任分配：各方过错程度
- 诉讼时效：是否超过诉讼时效

### 劳务合同纠纷
- 主体适格：合同双方身份及缔约能力
- 合同成立与生效：合同是否有效成立
- 合同内容：主要权利义务条款
- 违约事实：谁违反了什么义务
- 损害后果：违约造成的损失
- 违约责任：赔偿计算依据
- 诉讼时效：是否超过诉讼时效

### 民间借贷纠纷
- 主体适格：出借人与借款人身份
- 借贷合意：借款合同是否成立（借条/转账记录/聊天记录）
- 款项交付：是否实际支付借款
- 借款金额：本金及利息约定
- 还款情况：已还金额及时间
- 担保/保证：是否存在担保条款
- 诉讼时效：催款记录是否构成时效中断

### 买卖合同纠纷
- 主体适格：买卖双方身份
- 合同成立与生效：合同/订单/要约承诺
- 标的物：货物描述及规格
- 交付义务：是否交货、交货时间/地点
- 付款义务：是否付款、付款金额/时间
- 违约事实：逾期交货/逾期付款/质量不合格
- 损害后果：违约造成的损失
- 诉讼时效：是否超过诉讼时效

### 建设工程施工合同纠纷
- 主体适格：发包方、承包方、实际施工人
- 合同效力：是否存在挂靠/违法分包/转包
- 工程范围与内容：合同约定的施工范围
- 工程质量：是否竣工验收合格
- 工期：是否存在逾期完工
- 工程价款：合同价 vs 已付 vs 欠付
- 工程结算：结算文件是否确认
- 优先受偿权：是否主张及期限
- 诉讼时效/仲裁条款：争议解决途径

### 侵权责任纠纷（一般）
- 主体适格：侵权人与被侵权人
- 侵权行为：侵害行为的具体表现
- 损害后果：人身损害/财产损失
- 因果关系：行为与损害的关联
- 过错认定：故意/过失/无过错
- 免责/减轻事由：受害人过错、不可抗力等
- 赔偿范围：医疗费/误工费/精神损害等
- 诉讼时效：是否超过时效

### 房屋租赁合同纠纷
- 主体适格：出租方与承租方身份
- 合同成立与生效：租赁合同是否有效
- 租赁物：房屋地址、面积、用途
- 租金：约定金额、支付方式、押金
- 履行情况：交付/使用/维修义务
- 违约事实：逾期支付/擅自转租/提前解约
- 损害后果：违约造成的损失
- 诉讼时效：是否超过时效

**如果律师提供的案由不在上述列表中，根据案由名称自行推导法律构成要件。** 推导方法：从"主体—权利基础—事实—损害—因果关系—抗辩—时效"七个维度分析。

将拆解结果展示给律师，初始值全部为 null（表示尚未验证）。

## 步骤三：写入状态文件

### 生成 session_id

格式：`session_{YYYYMMDD}_{HHMMSS}`，使用当前时间。

### 写入 case-context.json

```json
{
  "case_type": "<案由>",
  "plaintiff": "<原告>",
  "defendants": ["<被告1>", "<被告2>"],
  "background": "<简要案情>",
  "lawyer_hypotheses": ["<假设1>", "<假设2>"],
  "lawyer_strategy_notes": ["<策略1>"],
  "focus_points": ["<关注点1>", "<关注点2>"],
  "legal_elements_checklist": {
    "<要件名>": null,
    ...
  },
  "updates": []
}
```

### 写入 case-state.json

```json
{
  "case_id": "<YYYY>-cv-<序号>",
  "case_name": "<原告> 诉 <被告> <案由>纠纷",
  "session_id": "<生成的 session_id>",
  "created_at": "<当前时间 ISO 格式>",
  "updated_at": "<当前时间 ISO 格式>",
  "phase": "phase_0_init",
  "phase_history": [],
  "evidence_count": 0,
  "evidence_read": 0,
  "evidence_approved": 0,
  "evidence_pending_review": 0,
  "evidence_unread": 0,
  "first_pass_completed": false,
  "cross_verify_completed": false,
  "report_generated": false,
  "report_finalized": false,
  "supplementary_rounds": 0,
  "report_version": null
}
```

### 初始化其他状态文件

- `file-manifest.json`：`{"files": []}`
- `reading-plan.json`：`{"groups": [], "reading_order_within_group": "by_date", "lawyer_confirmed": false}`
- `review-log.json`：`{"reviews": []}`

### 确保 workspace 目录结构存在

如果以下目录不存在则创建：
- `workspace/raw/`
- `workspace/processed/`
- `workspace/briefs/`
- `workspace/versions/`
- `workspace/meta/`

## 步骤四：输出确认

向律师输出以下确认信息：

```
已建立案件上下文（session_id: {session_id}）。

案件：{原告} 诉 {被告} {案由}纠纷
法律要件拆解：
  - {要件1}（{说明}）
  - {要件2}（{说明}）
  - ...

下一步：请将证据文件放入 workspace/raw/ 目录，然后执行 /preprocess 开始预处理。
```
