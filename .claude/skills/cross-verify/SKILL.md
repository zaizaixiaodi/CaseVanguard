# Skill：全局交叉验证

## 1. 四维验证方法论

### 维度一：时间线矛盾检测

从精要大合集中提取全部日期-事件对，构建案件时间线，检测以下异常：

| 检查类型 | 说明 | 示例 |
|----------|------|------|
| 同一事件不同日期 | 同一事实在不同证据中记载了不同日期 | E001竣工2025.07.01 vs E004完工2025.08.31 |
| 逻辑倒置 | 事件顺序违反因果关系 | 交付早于开工、付款早于签约 |
| 关键节点缺失 | 案件逻辑链中的必要环节无证据支撑 | 有合同无验收、有完工无移交 |
| 日期模糊 | 关键证据缺少明确日期 | 函件无落款日期、协议签署日模糊 |

**输出：** 时间线事件表（按日期排序）+ 矛盾列表（含严重性分级）

### 维度二：金额一致性校验

提取全部金额信息，按类别分组后交叉校验：

**金额类别：**
- A. 合同约定金额（合同总价、分期金额、质保金）
- B. 实际支付金额（已付款、转账记录）
- C. 欠付主张金额（欠款确认、催款金额）
- D. 关联金额（利息、违约金、损害赔偿）

**校验规则：**
1. 算术关系：合同价 - 已付 = 欠付（允许5%误差）
2. 分期校验：各期金额之和 = 合同总价
3. 金额来源一致性：同一金额在不同证据中的表述是否一致
4. 异常标注：大额资金去向不明、资金流转闭环（A→B→A）

**输出：** 金额校验表 + 不一致项 + 异常资金流向

### 维度三：主体一致性校验

提取全部主体及其在证据中的角色，构建主体角色矩阵：

**角色类型：** 签约方、建设单位、施工单位、付款方、收款方、发函方、收函方、担保方、保证人、实际控制人

**校验规则：**
1. 签约方vs履行方：合同甲方是否就是实际履行方
2. 付款方vs签约方：付款方与合同义务方是否一致
3. 收款方vs合同方：资金流入方与案件主体关系
4. 新主体追踪：未在合同中出现的主体，需标注其与案件的关系

**输出：** 主体角色矩阵 + 身份链矛盾 + 新主体清单

### 维度四：法律要件缺口检测

读取 `case-context.json` 的 `legal_elements_checklist`，逐一校验：

| 状态 | 定义 | 标注方式 |
|------|------|----------|
| ✅ 充分支撑 | 有直接证据证明该要件成立 | 列出支撑证据编号 |
| ⚠️ 部分支撑 | 有间接证据但存在缺口 | 标注缺什么 |
| ❌ 无证据 | 无任何证据触及该要件 | 建议补充方向 |

**校验方法：**
1. 读取每个要件的名称和描述
2. 在精要大合集中搜索与该要件相关的证据
3. 评估证据充分性：是否有直接陈述？是否需要推论？推论链条是否完整？
4. 对照律师 focus_points，检查关注点对应的要件是否有足够证据

**输出：** 要件覆盖表 + 缺口清单（含补充建议）

## 2. 矛盾严重性分级

| 级别 | 定义 | 对案件的影响 |
|------|------|-------------|
| 🔴 严重 | 直接影响核心诉求的事实矛盾 | 可能导致败诉或诉求被大幅削减 |
| 🟡 注意 | 影响赔偿范围或策略选择 | 不影响胜败但影响金额或路径 |
| ⚪ 信息 | 值得记录但不影响案件走向 | 文件瑕疵或OCR问题 |
