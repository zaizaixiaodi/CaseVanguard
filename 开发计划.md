# 卷宗先锋（CaseVanguard）开发计划

**版本：** v1.0
**创建日期：** 2026-05-12
**基于 PRD：** v0.2

---

## 开发纪律

**每个里程碑开始前，必须先重读 PRD 文档**（`项目需求和一些输入/PRD_卷宗先锋Agent_v0.2.md`），对照该阶段涉及的功能规格逐条确认，避免实现跑偏。具体做法：

1. 读取 PRD 中对应 Phase 的完整章节
2. 提取该阶段所有明确要求的字段、格式、流程节点
3. 对照任务清单，确认无遗漏
4. 开发过程中如对某个细节拿不准，回查 PRD 原文

---

## 开发总览

将开发拆分为 **10 个里程碑**，每个里程碑产出可独立验证的成果。按依赖关系排序，逐项推进。

## M1：项目脚手架搭建

**目标：** 建立完整的目录结构和核心配置文件，确保后续所有命令和技能有统一的运行基础。

### 任务清单

- [ ] **1.1 创建项目目录结构**
  - `.claude/skills/`、`.claude/scripts/`、`.claude/commands/`
  - `templates/`
  - `workspace/`（含 `raw/`、`processed/`、`briefs/`、`meta/`、`versions/`）
  - 初始为空目录结构，可通过 `.gitkeep` 或占位文件维持

- [ ] **1.2 编写 `claude.md`（Agent 主指令文件）**
  - 产品定位与角色说明
  - 行为红线（不做法律判断、不编造、不遗漏主体、不替代律师沟通）
  - 全局工作流程概述（Phase 0-4）
  - 状态文件读取规则（启动时从 `meta/` 重建上下文）
  - 输出物质量约束（压缩比、字数梯度、不冗余原则）
  - 错误处理与异常状态恢复

- [ ] **1.3 定义 JSON 状态文件 Schema**
  - `case-state.json`：全景状态（phase、evidence 统计、session_id）
  - `case-context.json`：案件上下文（案由、当事人、背景、策略、更新日志）
  - `file-manifest.json`：文件清单（证据编号、路径、状态）
  - `reading-plan.json`：分组方案（组ID、标签、证据列表、状态）
  - `review-log.json`：审批记录（时间、操作、内容）
  - 每个 Schema 写成模板文件放在 `templates/meta/` 下

- [ ] **1.4 编写状态管理工具脚本**
  - `.claude/scripts/state-manager.py`：统一的 JSON 状态文件读写工具
    - `read_state(filename)` → 返回解析后的 dict
    - `write_state(filename, data)` → 原子写入 + 自动更新 `updated_at`
    - `update_phase(new_phase)` → 自动追加 `phase_history`
    - `increment_counter(field)` → 更新计数字段

### 验证标准
- 目录结构完整，无遗漏
- `claude.md` 可被 Claude Code 正确加载
- 状态文件 Schema 符合 PRD 定义，字段完整
- `state-manager.py` 单元测试通过（读写、更新、原子性）

## M2：/init-case 命令

**目标：** 实现 Phase 0 案件初始化，律师通过对话式引导提供案件信息，Agent 建立完整的工作上下文。

### 任务清单

- [ ] **2.1 编写 `/init-case` 命令文件**（`.claude/commands/init-case.md`）
  - 检查是否已有未完成案件（读取 `case-state.json`）
  - 以对话方式引导律师输入：案由、当事人、案情、初步判断、关注点
  - 动态生成结构化引导表，不强制填满所有字段
  - 根据案由自动拆解法律构成要件 `legal_elements_checklist`

- [ ] **2.2 实现案件初始化逻辑**
  - 生成 `session_id`（格式 `session_{YYYYMMDD}_{HHMMSS}`）
  - 写入 `case-context.json`（初始信息 + 动态补充的 updates）
  - 写入 `case-state.json`（初始状态，phase = `phase_0_init`）
  - 创建 `workspace/` 完整目录结构（如尚未存在）
  - 输出法律要件初步拆解

- [ ] **2.3 实现上下文动态补充机制**
  - 律师在任意阶段可通过对话补充信息
  - Agent 自动追加到 `case-context.json` 的 `updates` 数组
  - 更新 `focus_points`、`lawyer_hypotheses` 等字段

### 验证标准
- 执行 `/init-case`，按引导表输入部分信息（模拟"暂不确定"场景），状态文件正确生成
- 案由为"提供劳务者受害责任纠纷"时，法律要件拆解覆盖 PRD 中列出的全部要件
- 再次执行 `/init-case` 时，能检测到已有未完成案件并提示

## M3：预处理能力 + /preprocess

**目标：** 实现 Phase 1 预处理流程，将原始证据文件转换为可阅读的 Markdown，并生成分组阅读方案。

**技术决策：**
- PDF/图片转 Markdown：复用已有 `mineru-ocr` skill，不单独编写转换脚本
- 大表压缩处理：推迟到 v2，v1 阶段对 Excel/CSV 做基础转换即可
- 重命名：在命令流程中直接实现，不单独编写脚本

### 任务清单

- [ ] **3.1 编写预处理技能**（`.claude/skills/preprocess.md`）
  - 文件扫描规则（遍历 `raw/` 目录，识别格式）
  - 格式判断逻辑（文本型 PDF vs 扫描型 PDF vs 图片 vs Excel 等）
  - 分组规则（类型关联性 + 硬证据优先 + 律师关注点驱动排序）
  - 重命名规则
  - 调用 `mineru-ocr` skill 进行格式转换

- [ ] **3.2 编写 `/preprocess` 命令文件**（`.claude/commands/preprocess.md`）
  - Step 1.1：扫描 raw/ 生成 file-manifest.json
  - Step 1.2：逐文件格式转换（调用 mineru-ocr skill）
  - Step 1.3：统一重命名
  - Step 1.4：生成分组建议并呈报律师确认

- [ ] **3.3 实现分组建议算法**
  - 按证据类型分组（合同类、转账类、沟通记录类等）
  - 硬证据组优先于软证据组
  - 同组内按日期排序
  - 结合律师关注点调整优先级（关注点关键词匹配）
  - 输出分组方案含理由说明

### 验证标准
- 准备多种格式的测试文件（PDF、图片、Word、TXT），放入 `raw/`
- 执行 `/preprocess`，检查：file-manifest.json 正确记录所有文件、processed/ 目录下全部转为 MD
- 分组建议合理，包含理由说明
- 律师调整分组后，reading-plan.json 正确更新

## M4：/read-group + /read-next + 精要模板

**目标：** 实现 Phase 2 核心阅读流程，Agent 逐份阅读证据并按模板生成结构化精要。

### 任务清单

- [ ] **4.1 编写阅读技能**（`.claude/skills/read-evidence.md`）
  - 兜底七要素摘录规则（主体、意思表示、效力、权利义务、失衡事实、抗辩事由、金额）
  - "法庭试金石"判断法说明
  - 不摘录内容的排除规则
  - 大表精要的特殊处理规则
  - 梗概撰写规范（3-5 句话，客观事实）

- [ ] **4.2 编写《证据精要》模板**（`templates/evidence-brief.md`）
  - 元信息区（编号、文件名、类型、形式、页数、可信度初判）
  - 梗概区
  - 关键细节摘录表（序号、要素类型、内容摘录、原文锚点）
  - 初步发现与疑问区

- [ ] **4.3 编写 `/read-group` 命令**（`.claude/commands/read-group.md`）
  - 读取 `reading-plan.json` 获取当前组的证据列表
  - 按组内顺序逐份阅读，每份生成精要写入 `briefs/`
  - 静默连续阅读模式：组内不逐份暂停
  - 中断条件：发现矛盾 → 暂停并记录到 reading-plan.json
  - 组完成后打包呈报律师

- [ ] **4.4 编写 `/read-next` 命令**（`.claude/commands/read-next.md`）
  - 支持指定 E编号 阅读特定证据
  - 无参数时自动读取下一份未读证据
  - 阅读后生成精要并更新状态

- [ ] **4.5 实现原文锚点机制**
  - 精要中每条摘录记录对应的 `processed/E###.md` 行号范围
  - 格式：`E001.md#L45-L52`
  - 律师/下游 Agent 可据此回溯原文

### 验证标准
- 准备一组测试证据（至少 3 份，含合同类和聊天记录类）
- 执行 `/read-group 1`，检查：
  - 每份证据精要覆盖兜底七要素
  - 精要格式与模板一致
  - 原文锚点记录正确
  - file-manifest.json 中 `reading_status` 更新
  - 梗概控制在 3-5 句话内

## M5：/approve + /revise + 精要大合集

**目标：** 实现审批流程和精要汇总机制，通过审批的精要入库到大合集。

### 任务清单

- [ ] **5.1 编写 `/approve` 命令**（`.claude/commands/approve.md`）
  - 无参数：通过当前组所有精要
  - 指定排除：`/approve 但 E005 需要修改`
  - 通过后：精要追加到 `evidence-collection.md`，file-manifest.json 状态更新为 `approved`
  - 更新 `case-state.json` 计数

- [ ] **5.2 编写 `/revise` 命令**（`.claude/commands/revise.md`）
  - 接受律师的修改意见
  - 重新加载原文 + 结合修改意见重新生成精要
  - 提交律师再次审批

- [ ] **5.3 编写《精要大合集》模板**（`templates/evidence-collection.md`）
  - 按分组排列已通过的精要
  - 每组前附组标签说明
  - 末尾附压缩比自检报告

- [ ] **5.4 实现精要入库逻辑**
  - 追加到 `evidence-collection.md`
  - 更新 `review-log.json`（记录审批时间、操作）
  - 更新 `case-state.json`（evidence_approved 计数）
  - 自动检测是否所有组完成 → 标记 `first_pass_completed`

- [ ] **5.5 实现压缩比自检**
  - 计算 `processed/` 下所有 MD 文件总大小
  - 计算 `evidence-collection.md` 大小
  - 输出压缩比（目标 ≤ 1/4，理想 1/10）
  - 压缩比不足 1/4 时提示自查

### 验证标准
- 完成一组阅读后，执行 `/approve`，精要正确入库到大合集
- 执行 `/revise E001 补充第3页的违约金条款`，精要重新生成并通过审批
- 压缩比计算正确，能在不足时发出提示

## M6：即时交叉验证

**目标：** 在阅读过程中实现轻量级增量比对，发现矛盾时即时中断并提醒律师。

### 任务清单

- [ ] **6.1 编写即时交叉验证逻辑**（集成到 `read-evidence.md` 技能中）
  - 每完成一份精要，自动触发比对
  - 检查项：日期/事件冲突、金额不一致、主体信息矛盾
  - 比对对象：已入库精要（`evidence-collection.md`）+ `timeline.md`

- [ ] **6.2 实现矛盾检测算法**
  - 提取当前精要中的日期、金额、主体信息
  - 与已有精要逐一比对
  - 定义"明显矛盾"的阈值（如同一事项金额差异 > 10%）
  - 输出矛盾描述 + 涉及的证据编号

- [ ] **6.3 实现中断与恢复机制**
  - 矛盾触发时：中断静默阅读，向律师汇报
  - 记录暂停位置到 `reading-plan.json`（paused_at、pause_reason）
  - 精要中标注 `⚠️ 交叉验证警告`
  - 律师回应后从暂停处继续

### 验证标准
- 准备含矛盾信息的测试数据（如两份证据中金额不一致、日期冲突）
- 阅读过程中自动检测到矛盾并中断
- 中断后暂停位置和原因正确记录
- 律师回复后继续阅读后续证据

## M7：/cross-verify 全局交叉验证

**目标：** 实现 Phase 3 全局交叉验证，四维系统性排查。

### 任务清单

- [ ] **7.1 编写交叉验证技能**（`.claude/skills/cross-verify.md`）
  - 时间线矛盾检测：通读所有精要中的日期与事件，检查顺序合理性和同日冲突
  - 金额一致性校验：合同约定 vs 实际转账 vs 当事人主张
  - 主体一致性校验：签约主体 vs 履行主体 vs 转账主体
  - 法律要件缺口检测：基于 `legal_elements_checklist` 检查证据覆盖

- [ ] **7.2 编写 `/cross-verify` 命令**（`.claude/commands/cross-verify.md`）
  - 读取 `evidence-collection.md` + `timeline.md` + `case-context.json`
  - 执行四维验证
  - 输出验证结果汇总报告
  - 对可自动解决的矛盾（通过重读原文），自动执行修复

- [ ] **7.3 实现法律要件缺口分析**
  - 读取 `case-context.json` 中的 `legal_elements_checklist`
  - 遍历每个要件，检查精要中是否有证据支撑
  - 输出缺口清单（含严重程度评估）
  - 将缺口写入交付物的"风险提示"部分

- [ ] **7.4 实现自动修复与标注**
  - 矛盾可通过重读原文解决 → 自动重读并更新精要
  - 矛盾无法解决 → 标注后写入"风险提示"
  - 缺口 → 写入"下一步行动计划"
  - 更新 `case-state.json`（`cross_verify_completed = true`）

### 验证标准
- 准备含多种矛盾的测试数据集（时间线冲突、金额不一致、主体混淆、要件缺口）
- 执行 `/cross-verify`，四维验证全部触发并正确报告
- 可自动修复的矛盾被正确修复
- 法律要件缺口分析覆盖 PRD 中示例的所有条目

## 测试策略

每个里程碑完成后，执行以下验证：

1. **单元验证：** 该里程碑的功能独立可用
2. **集成验证：** 与前序里程碑串联执行完整流程
3. **边界验证：** 空输入、异常状态、并发场景
4. **端到端验证：** M8 完成后，使用 PRD 第十二节的典型使用流程做全链路测试
