**产品名称：** 卷宗先锋（CaseVanguard）

## 一、产品定位与命名

### 1.1 产品定位

"卷宗先锋"是一个运行在 Claude Code CLI 环境中的 AI Agent，采用 **folder-as-an-app** 架构。它承担民事诉讼中原告代理律师**第一轮证据审查与案情梳理**的工作，将律师从海量、碎片化、主观化的原始证据材料中解放出来，输出高度压缩的结构化文档，作为后续诉讼全流程的基础。

本项目本质上是一个**脚手架（scaffold）**：每个新案件复制一份 folder 即可启动，案件完成后可清理工作区以复用。

**核心价值主张：** 20 份证据、300KB 文本 → 一份精控字数的《案件初探精要》+ 一份可长期翻阅的《精要大合集》。

### 1.2 命名

| 中文名 | 英文标识 | 说明 |
|--------|----------|------|
| **卷宗先锋** | **CaseVanguard** | 强调"打头阵"的角色，是整个诉讼 Agent 体系的第一棒 |

## 二、用户角色与协作模型

### 2.1 角色定义

| 角色 | 身份 | 职责 |
|------|------|------|
| **律师（Human）** | 操作者、决策者 | 提供案件背景与策略、拆分证据、确认分组与阅读方案、审批精要、最终定稿 |
| **卷宗先锋 Agent（AI）** | 执行者、辅助分析者 | 预处理文件、逐份阅读证据、输出精要、即时与全局交叉验证、生成交付物 |
| **上层诉讼 Agent（AI，可选）** | 调度者 | 通过 CLI + session ID 调用卷宗先锋，获取交付物，传递给下游 Agent |

### 2.2 人机协作模型

```
律师提供案件背景 + 拆分好的证据文件
        │
        ▼
  ┌──────────────────────────┐
  │  Phase 0: 案件初始化       │  律师输入背景、案由、初步策略（动态补充）
  │  建立案件上下文            │
  └───────────┬──────────────┘
              ▼
  ┌──────────────────────────┐
  │  Phase 1: 预处理           │  Agent 独立完成
  │  格式化、降噪              │
  │  → Agent 建议分组+阅读顺序 │  律师确认或调整
  └───────────┬──────────────┘
              ▼
  ┌──────────────────────────┐
  │  Phase 2: 分组阅读         │  按分组静默阅读 → 组内遇疑暂停 → 组完成后提交律师审批
  │  输出《证据精要》          │  每读一份即时交叉验证，发现矛盾即时提醒
  └───────────┬──────────────┘
              ▼
  ┌──────────────────────────┐
  │  Phase 3: 全局交叉验证     │  Agent 通读精要大合集，全面核查
  └───────────┬──────────────┘
              ▼
  ┌──────────────────────────┐
  │  Phase 4: 输出交付物       │  Agent 生成 → 律师修改 → 定稿
  │  《案件初探精要》          │
  └──────────────────────────┘
```

> - **Skills**（`.claude/skills/<name>/SKILL.md`）：目录式结构，SKILL.md 为入口文件。适合需要辅助文件、模板、或复杂 frontmatter 的功能。每个 skill 自动注册为 `/<name>` 可调用命令。

> - 详细的 frontmatter 字段（`description`、`allowed-tools`、`disable-model-invocation` 等）参见 [Claude Code Skills 官方文档](https://docs.anthropic.com/en/docs/claude-code/skills)。

### 3.2 全景状态管理

**case-state.json** — 案件级别的全景状态，记录整个流程的宏观进度：

```json
{
  "case_id": "2026-cv-001",
  "case_name": "张三诉李四、XX体育公司提供劳务者受害责任纠纷",
  "session_id": "session_20260512_143000",
  "created_at": "2026-05-12T14:30:00",
  "updated_at": "2026-05-12T18:45:00",
  "phase": "phase_2_reading",
  "phase_history": [
    { "phase": "phase_0_init", "completed_at": "2026-05-12T14:35:00" },
    { "phase": "phase_1_preprocess", "completed_at": "2026-05-12T15:10:00" }
  ],
  "evidence_count": 12,
  "evidence_read": 7,
  "evidence_approved": 5,
  "evidence_pending_review": 2,
  "evidence_unread": 5,
  "first_pass_completed": false,
  "cross_verify_completed": false,
  "report_generated": false,
  "report_finalized": false,
  "supplementary_rounds": 0,
  "report_version": null
}
```

**要点说明：**

- `session_id`：唯一标识当前案件的工作会话。上层诉讼 Agent 可通过此 ID 唤醒卷宗先锋。恢复上下文不依赖重放对话历史，而是通过读取 `meta/` 下的状态文件重建认知，从而控制 token 消耗。
- `phase`：当前所处阶段（`phase_0_init` → `phase_1_preprocess` → `phase_2_reading` → `phase_3_cross_verify` → `phase_4_deliverable` → `completed`）。
- `supplementary_rounds`：追加证据或重新阅读的累计次数，便于追踪案件经历了几轮补充。
- `first_pass_completed`：所有证据是否至少完成过一轮阅读。

## 四、详细功能规格

### Phase 0：案件初始化

**触发方式：** `/init-case`

**流程：**

1. Agent 生成一个结构化引导表（基于 `case-context.json` 的字段），以自然对话方式询问律师。
2. 律师提供能提供的信息：案由、当事人、基本案情、初步假设、关注重点。**不强制填满**，允许"暂不确定"。
3. Agent 根据律师的输入和案由，初步拆解法律构成要件（`legal_elements_checklist`），作为后续摘录和缺口检测的参照。
4. 律师可补充策略性判断（如"我怀疑被告方存在挂靠关系"、"重点关注安全责任分配"）。
5. Agent 将所有信息写入 `case-context.json`，生成 `case-state.json`（含 `session_id`），创建 `workspace/` 目录结构。

**关键原则：** 案件上下文是活的文档。律师在任何阶段都可以通过自然对话补充信息，Agent 自动追加到 `case-context.json` 的 `updates` 中。

---

### Phase 1：预处理（Preprocess）

**触发方式：** `/preprocess`

**输入：** 律师将拆分好的证据文件放入 `workspace/raw/` 目录。

**处理流程：**

**Step 1.1 — 文件扫描与清单生成**

扫描 `raw/` 目录，对每个文件记录：文件名、格式、大小、修改时间。生成 `file-manifest.json`。

**Step 1.2 — 格式转换（"降噪"）**

| 原始格式 | 转换目标 | 方法 |
|----------|----------|------|
| PDF（文本型） | Markdown | 调用外部 PDF-to-MD API，保留表格结构 |
| PDF（扫描型/图片型） | Markdown | 先 OCR 再转 MD |
| 图片（.jpg/.png） | 文字描述 + OCR 文本 | 调用 OCR 工具提取文字；照片类生成文字描述 |
| 音频（.mp3/.wav/.m4a） | 文字稿 | 调用语音转写 API，输出带时间戳的文字稿（**v2 实现，v1 由律师提供文字稿**） |
| 视频（.mp4/.mov） | 文字稿 + 关键帧 | **v2 实现，v1 由律师提供文字稿或描述** |
| Excel/CSV（大表） | 压缩摘要 + 原始保留 | 见 Step 1.3 |
| Word/TXT | Markdown | 直接转换 |

**Step 1.3 — 大表预处理（Token 节约策略）**

对于超过 100 行的表格（如转账明细、打卡记录）：

1. 保留表头 + 前 5 行样本，让 Agent 理解表结构。
2. 按"交易主体"或"关键字段"进行分组统计（如：共 1000 行，涉及主体 A 的有 300 行，涉及主体 B 的有 50 行，其余 650 行涉及无关第三方）。
3. 输出压缩摘要文件（`E007_summary.md`），同时保留原始文件引用，标注"如需查看原始数据，见 `processed/E007_full.csv`"。
4. 律师可以在审阅时指定"展开某主体的所有记录"。

**Step 1.4 — 统一重命名**

按规则重命名：`{序号}_{证据类型}_{日期}_{简述}.{ext}`

示例：`01_合同类_20220501_合作协议.pdf`、`02_转账类_20220615_招行转账回单.pdf`

Agent 初步推断证据类型和日期，无法判断时标注 `00_待分类_未知日期_原文件名`，后续由律师确认。

**Step 1.5 — 分组与阅读顺序建议（新增）**

Agent 根据以下规则生成分组建议：

1. 按证据类型和关联性划分为若干组（通常 3-6 组）。
2. 硬证据组排在软证据组之前，同组内按日期排序。
3. **根据案由和律师在 Phase 0 提供的关注点，主动建议优先阅读的证据。** 例如人身损害案件中，如果律师特别关注安全责任归属，Agent 可能建议将"录音文字稿"提前到第一组阅读（因为可能包含对方承认过错的关键表述），而非机械地排在所有硬证据之后。
4. 每组标注建议的阅读模式：`silent`（静默连读，完成后统一审批）。

Agent 将分组方案和理由呈报律师，**律师确认或调整后**写入 `reading-plan.json`。

**输出：** 预处理结果清单 + 分组阅读方案建议。律师确认后进入 Phase 2。

---

### Phase 2：分组阅读与精要输出

**触发方式：** `/read-group [组号]`（按分组阅读）或 `/read-next [E编号]`（读指定/下一份）

#### 2.1 分组静默阅读模式

**默认流程（静默模式）：**

1. Agent 按当前组的阅读队列，逐份阅读证据并生成《证据精要》。
2. **组内连续阅读，不逐份暂停等待审批**——除非遇到以下情况需要中断：
   - 发现当前证据与已入库的精要存在**明显矛盾**（即时交叉验证触发，见 2.4）
   - 证据内容**严重模糊或残缺**，无法提取有效信息
   - Agent 对摘录取舍**拿不准**，需要律师的法律判断
3. 一组读完后，将该组所有精要打包提交律师审批。
4. 律师可以逐份审阅，也可以通读后统一 `/approve`。对有问题的个别精要 `/revise`。

**中断恢复：** 如果组内某份证据触发了暂停，Agent 记录暂停位置和原因到 `reading-plan.json`。律师回应后，Agent 从暂停处继续。

#### 2.2 阅读规则

Agent 阅读每份证据的 MD 化文本时，执行以下判断逻辑：

**必须摘录的内容（兜底七要素）：**

| 序号 | 要素 | 说明 | 示例 |
|------|------|------|------|
| ① | 主体信息 | 合同各方、保证人、第三方、实际控制人；转账记录中的付款方/收款方 | "甲方：A公司；乙方：B，身份证号xxx；保证人：C" |
| ② | 意思表示与权利义务变动行为 | 合同签订、付款、催告、承诺、自认、变更协议等 | "B 在聊天中自认与承包方系挂靠关系" |
| ③ | 影响法律关系效力的事项 | 可能导致合同无效、可撤销、效力待定的情节 | "挂靠可能导致施工合同无效" |
| ④ | 权利义务条款 | 与本案请求权基础直接相关的合同条款 | "第5条：付款节点为竣工验收后30日内" |
| ⑤ | 导致权利义务失衡的事实 | 违约行为、侵权行为、损害事实 | "A 推倒 B 致 B 右臂骨折" |
| ⑥ | 抗辩事由 | 对方可能提出的抗辩或免责理由 | "原告在事发前存在违规操作" |
| ⑦ | 金额相关 | 合同价款、实际付款、损失金额、赔偿计算依据 | "合同总价 50 万，已付 30 万，欠付 20 万" |

**"法庭试金石"判断法（补充规则）：** 如果 Agent 仍然不确定某个细节是否该摘录，则设想：如果此案上了法庭，这个细节会不会成为双方掰扯的焦点？我方代理词或起诉状中会不会死咬这一点？对方会不会拿这一点咬死我方？如果答案是"可能会"，则必须摘录。

**不摘录的内容：**

- 与案件无关的日常寒暄、客套内容
- 大表中与案件主体/争议无关的行项
- 合同中的通用格式条款（如"本合同一式两份"等程序性条款），除非对争议有影响
- 重复出现的同类信息（已摘录一次的不再重复，但标注重复次数和范围）

**大表的特殊处理：**

如果一个大表中有大量行指向同一交易主体或同一类型交易（如 300 行均为与被告的转账），不逐行摘录，而是输出统计摘要并标注疑问：

> "该转账明细共 1000 行。其中 300 行（行号 45-120、200-350、500-574）的收款方均为被告 B，合计金额 ¥823,000。建议律师确认：这些转账是否均与本案争议相关？是否存在资金过桥或财务混同的情况？"

#### 2.4 即时交叉验证（新增）

**触发时机：** Agent 每完成一份证据的阅读，自动将新提取的关键信息（主体、日期、金额、事实主张）与已入库的所有精要进行轻量级比对。

**检查项：**
- 本份证据中的日期/事件是否与已有时间线冲突
- 本份证据中的金额是否与已有金额记录不一致
- 本份证据中出现的主体是否与已有主体信息矛盾

**处理方式：**
- 发现矛盾 → Agent 在当前证据的精要中标注 `⚠️ 交叉验证警告`，并中断静默阅读，向律师汇报。
- 未发现矛盾 → 静默继续。

**注意：** 即时交叉验证是轻量级的"增量比对"，不替代 Phase 3 的全局交叉验证。全局验证会做更全面的系统性排查。

#### 2.5 审批流程

**组内阅读完毕后：**

1. Agent 将该组所有精要呈报律师（列出梗概清单 + 标注有疑问的精要）。
2. 律师选择：
   - `/approve`：该组全部通过，入库到 `evidence-collection.md`。
   - `/approve` + 指定排除某份：如"approve，但 E005 需要修改"。
   - `/revise E005 [意见]`：针对某份提出修改。
3. 通过的精要追加到 `evidence-collection.md`，状态标记为 `approved`。

### Phase 3：全局交叉验证

**触发方式：** 所有组阅读完毕后自动提示，或律师手动 `/cross-verify`

**输入：** `evidence-collection.md`（精要大合集）+ `timeline.md` + `case-context.json`

**验证维度：**

**3.1 时间线矛盾检测**

通读所有精要中提取的日期与事件，检查：
- 时间顺序是否合理（如：付款日期早于合同签订日期？）
- 同一日期的不同证据是否冲突（如：打卡记录显示正常上班，但医院记录显示同日住院？）

**3.2 金额一致性校验**

- 合同约定金额 vs 实际转账金额 vs 当事人主张金额是否一致
- 多份转账记录的加总是否与当事人陈述吻合

**3.3 主体一致性校验**

- 合同签约主体 vs 实际履行主体 vs 转账主体是否一致
- 聊天记录中的昵称/微信号是否能关联到合同当事人

**3.4 法律要件缺口检测**

基于 `case-context.json` 中的 `legal_elements_checklist`，拆解法律构成要件，检查证据覆盖情况：

| 要件 | 需要证明的事项 | 现有证据支撑 | 缺口 |
|------|---------------|-------------|------|
| 主体适格 | 原被告身份、合同关系 | E001合同、E002身份证复印件 | ⚠️ 被告公司工商信息未提供 |
| 权利基础 | 合同有效成立 | E001合同 | ⚠️ 仅有复印件 |
| 违约/侵权事实 | 损害发生 | E003派出所回执、E004现场照片 | ⚠️ 缺行政处罚决定书 |
| 损害后果 | 损失金额 | E005医院发票 | ⚠️ 缺用药清单 |
| 因果关系 | 违约/侵权与损害的因果 | E004视频（模糊） | ⚠️ 视频质量差，证明力不足 |
| 诉讼时效 | 未超时效/时效中断 | E006催款聊天记录 | ✅ 暂无明显问题 |

**3.5 其他一致性校验（可扩展）**

此列表在具体项目中根据案件类型补充，例如：
- 证人陈述之间的一致性
- 当事人前后陈述的一致性
- 物证与言辞证据的对应关系

**输出：** 向律师汇报交叉验证结果。如发现矛盾或缺口：
- 矛盾可通过重新阅读原文解决的 → Agent 自动重读并更新精要
- 矛盾无法解决或需要律师判断的 → 标注后写入交付物的"风险提示"部分
- 证据缺口 → 写入交付物的"下一步行动计划"

## 五、《精要大合集》体量约束（新增）

`evidence-collection.md` 是所有证据精要的汇总，是律师可长期翻阅修改的中间文稿。

**压缩约束：**

- **目标：** 总体量不超过原始证据 MD 文档总体量的 **1/4**，理想状态 **1/10**。
- **实现方式：** 不设硬性字数上限，而是通过以下规则自然控制体量：
  1. 精要中**不允许整段复制原文**，每条摘录都必须是提炼后的表述。
  2. 同类信息只摘录一次，后续出现标注"同上，重复 X 次"。
  3. 大表类证据只放压缩摘要，不放明细。
  4. 梗概部分控制在 3-5 句话以内。
- **边界情况：** 如果原始证据本身信息密度极高（如 5 份证据总共 30KB），则允许精要接近原文体量，只要没有冗余。核心标准是**"不冗余"**，而非机械压缩比。
- **自检机制：** Agent 在汇总完成后，自动计算压缩比并报告。如果压缩比不足 1/4，Agent 应自查是否存在不必要的冗余，并提示律师。

## 八、Session 与上层 Agent 调度

### 8.1 Session ID 机制

每次 `/init-case` 时生成唯一的 `session_id`（格式：`session_{日期}_{时间}`），写入 `case-state.json`。

**上层 Agent 调度流程：**

1. 上层诉讼 Agent 通过 CLI 启动卷宗先锋，传入 `session_id`。
2. 卷宗先锋启动后，首先读取 `workspace/meta/case-state.json`，根据 `phase` 字段判断当前进度。
3. 进一步读取 `case-context.json`、`file-manifest.json`、`reading-plan.json` 等状态文件，重建对案件的认知。
4. **不依赖重放对话历史。** Token 消耗仅限于读取状态文件（通常几千 token），而非整个对话历史。

**上层 Agent 可用的查询接口：**

- `/status` → 返回 `case-state.json` 的关键字段，上层 Agent 据此判断是否需要介入。
- 读取 `case-probe-report.md` → 获取最终交付物，传递给下游 Agent。

### 8.2 多 Session 场景

如果律师在一个案件中需要多次回到卷宗先锋（如追加证据、修改精要），每次通过 `session_id` 唤醒时，Agent 都从状态文件重建上下文。多次会话之间的连续性通过文件系统保证，而非对话记忆。

## 十一、质量约束与注意事项

### 11.1 Agent 行为红线

- **不做法律判断。** Agent 只提取事实、发现矛盾、标注缺口。"这份合同是否有效"这类判断由律师完成。Agent 可以提示"该条款可能涉及效力问题，建议律师关注"，但不下结论。
- **不编造内容。** 所有摘录必须忠实于原文。如果原文模糊，如实标注"原文表述模糊，无法确定含义"。
- **不遗漏主体。** 每份证据中出现的所有人名、公司名、账号等主体标识，都必须记录，哪怕暂时看不出与案件的关联。
- **不替代律师与当事人沟通。** Agent 的所有产出面向律师，不直接面对当事人。

### 11.2 输出物体量控制

| 输出物 | 体量约束 | 说明 |
|--------|---------|------|
| 单份《证据精要》 | 无硬性限制 | 以"摘录完关键信息"为准，但不允许整段复制原文 |
| 《精要大合集》 | ≤ 原始 MD 总量的 **1/4**，理想 **1/10** | 边界情况：原始材料信息密度极高时，允许接近原文体量，但不允许冗余 |
| 《案件初探精要》正文 | 简单 1000-1500 / 中等 1500-2000 / 复杂 2000-3000 字 | 复杂案件必须附 ≤200 字的 Executive Summary |
| 关键细节速查表 | 不计入正文字数 | 每条 1-2 句话，包含原文锚点 |

### 11.3 数据安全

- 所有案件材料仅在本地 `workspace/` 目录中处理，不上传到非必要的第三方服务。
- 调用外部 API（如 OCR、语音转写）时，应优先使用支持私有化部署的方案。
- 律师应在使用前确认外部 API 的数据隐私政策是否符合律所合规要求。

## 十三、后续扩展方向（Out of Scope for v1）

以下能力不在当前版本范围内，但属于自然的演进方向：

1. **下游 Agent 对接：** 基于《案件初探精要》自动生成起诉状草稿、正式证据目录。
2. **法律知识库集成：** 对接法律法规数据库，自动匹配相关法条和判例。
3. **多案件关联：** 同一律师处理的多个案件之间的证据关联分析。
4. **当事人沟通模板：** 根据行动计划自动生成给当事人的"作业清单"邮件。
5. **被告方视角模拟：** 从被告角度审视证据，预演可能的抗辩策略。
6. **音视频处理：** 音频转写、视频关键帧提取（v2）。
